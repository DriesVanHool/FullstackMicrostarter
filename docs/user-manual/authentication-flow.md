# Authentication flow

The current authentication flow is initiated by the hosted React apps in the browser and handled by Keycloak.

## Current flow

1. The portal or workspace frontend initializes the Keycloak JavaScript adapter
2. Unauthenticated users see the local login screen for the app they opened and start sign-in with Keycloak
3. Keycloak authenticates the user with Authorization Code flow and PKCE
4. The browser returns to the hosted frontend with an authenticated session
5. The client keeps the authenticated session available through the shared auth package
6. Gateway and service-level security validate JWTs and derive authorities from `realm_access.roles`

## Current user experience

- unauthenticated users see a login page in the app they opened
- the app login page sends sign-in to Keycloak and offers a Keycloak-hosted password reset entry point
- authenticated users can reach the portal launcher and the workspace
- sign-out redirects back through Keycloak logout

## Password setup and recovery

The starter platform deliberately keeps credential flows in Keycloak instead of re-implementing them in app code.

- first-time user setup happens through the required-action email triggered when an admin creates a user from Portal admin user management
- password recovery happens in the hosted Keycloak flow
- the reset email link is generated by Keycloak and uses a Keycloak-managed action token
- the frontend does not post passwords to backend APIs for reset purposes
- the backend does not mint app-specific password-reset tokens for this flow

## Current behavior details

- the shared auth provider preserves the current route during Keycloak auth redirects instead of always bouncing users back to app root
- login UI branding is controlled partly in the frontend apps and partly in the hosted Keycloak theme
